
# 🧹 Nexus Docker Cleaner

Скрипт для автоматической очистки Docker-репозиториев в **Nexus 3** по правилам хранения образов: сроку жизни (*retention*) и количеству резервных версий (*reserved*).

---

## 📌 Описание

Этот скрипт подключается к REST API Nexus 3 и удаляет устаревшие компоненты (образы), основываясь на следующих параметрах.

---

## ⚙️ Настройки

Перед запуском необходимо задать переменные окружения (можно через `.env`):

```env
USER_NAME=your-nexus-username
PASSWORD=your-nexus-password
BASE_URL=https://your-nexus-domain/
DRY_RUN=true   # true — ничего не удаляет, только логирует
```

---

## 🔧 Параметры

### 🟢 `retention` (в днях)
Сколько времени можно хранить образы. Это ограничение по максимальному возрасту.

Если образ старше retention и он не входит в список "зарезервированных", он будет удалён.

> Пример: если `retention = 8`, то всё старше 8 дней может быть удалено.

---

### 🔵 `reserved` (по количеству)
Сколько самых свежих версий нужно обязательно сохранить. Даже если они старше retention, они сохраняются.

> Пример: если `reserved = 1`, то самая свежая версия с данным префиксом и именем не удаляется, даже если ей 100 дней.

---

## 🧩 Дополнительные параметры

### 🔹 `MAX_RETENTION_DAYS`
**Что делает:**  
Это глобальный предел возраста любой версии (с префиксом или без).

Если включён и компоненту больше MAX_RETENTION, он будет удалён в любом случае, даже если он входит в резерв или не превышает retention по префиксу.

> **Пример:**
> ```
> MAX_RETENTION_DAYS=30
> ```
> Образу `r1-build1:version-5` уже 35 дней  
> ➡️ Он будет удалён, даже если входит в 1 резерв по `r1-`.

**Можно отключить:**  
Просто не указывайте переменную `MAX_RETENTION_DAYS` в `.env`.

---

### 🔹 `NO_PREFIX_RETENTION_DAYS`
**Что делает:**  
Применяется только к тегам **без префикса**.  
Если у версии нет префикса (например, `v1.0.0`, `123`, `build42`), и `NO_PREFIX_RETENTION_DAYS` включён, образы старше указанного срока будут удалены.

> **Пример:**
> ```
> NO_PREFIX_RETENTION_DAYS=15
> ```
> Образ `build-123`, ему 20 дней  
> ➡️ Он попадёт под удаление

**Можно отключить:**  
Не указывайте `NO_PREFIX_RETENTION_DAYS` в `.env`.  
➡️ Тогда образы без префикса не будут удаляться по времени, только по `MAX_RETENTION`, если он включён.

---

## 🤔 Поведение при отсутствии параметров

Если **оба параметра отключены** (`MAX_RETENTION_DAYS` и `NO_PREFIX_RETENTION_DAYS`), скрипт опирается только на `PREFIX_RULES`.

Если `prefix_rules` отсутствует в YAML-конфиге:
- Все теги считаются "без префикса".
- К ним применяется `no_prefix_retention_days`, если он задан.
- Если и он не указан — такие образы не удаляются вообще.

---

## 🛠 Пример конфигурационного YAML-файла

```yaml
repo_names:
  - test1
  - test2

prefix_rules:
  dev:
    retention_days: 0
    reserved: 1
  test:
    retention_days: 0
    reserved: 1

max_retention_days: 30        # Можно опустить
no_prefix_retention_days: 15  # Можно опустить
dry_run: true                 # true — только лог, без удаления
```
