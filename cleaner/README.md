
# 🧹 Nexus Docker Cleaner

Скрипт для автоматической очистки Docker-репозиториев в **Nexus 3** по правилам хранения образов: сроку жизни (*retention*) и количеству резервных версий (*reserved*).

---

## 📌 Описание

Этот скрипт подключается к REST API Nexus 3 и удаляет устаревшие компоненты (образы), основываясь на следующих параметрах.

---

## ⚙️ Настройки

Перед запуском необходимо задать переменные окружения (можно через `.env`):

```env
USER_NAME=your-nexus-username
PASSWORD=your-nexus-password
BASE_URL=https://your-nexus-domain/
DRY_RUN=true   # true — ничего не удаляет, только логирует
```

---

## 🔧 Параметры

### 🟢 `retention` (в днях)
Сколько времени можно хранить образы. Это ограничение по максимальному возрасту.

Если образ старше retention и он не входит в список "зарезервированных", он будет удалён.

> Пример: если `retention = 8`, то всё старше 8 дней может быть удалено.

---

### 🔵 `reserved` (по количеству)
Сколько самых свежих версий нужно обязательно сохранить. Даже если они старше retention, они сохраняются.

> Пример: если `reserved = 1`, то самая свежая версия с данным префиксом и именем не удаляется, даже если ей 100 дней.

---

## 🧩 Дополнительные параметры

### 🔹 `MAX_RETENTION_DAYS`
**Что делает:**  
Это глобальный предел возраста любой версии (с префиксом или без).

Если включён и компоненту больше MAX_RETENTION, он будет удалён в любом случае, даже если он входит в резерв или не превышает retention по префиксу.

> **Пример:**
> ```
> MAX_RETENTION_DAYS=30
> ```
> Образу `r1-build1:version-5` уже 35 дней  
> ➡️ Он будет удалён, даже если входит в 1 резерв по `r1-`.

**Можно отключить:**  
Просто не указывайте переменную `MAX_RETENTION_DAYS` в `.env`.

---

### 🔹 `NO_PREFIX_RETENTION_DAYS`
**Что делает:**  
Применяется только к тегам **без префикса**.  
Если у версии нет префикса (например, `v1.0.0`, `123`, `build42`), и `NO_PREFIX_RETENTION_DAYS` включён, образы старше указанного срока будут удалены.

> **Пример:**
> ```
> NO_PREFIX_RETENTION_DAYS=15
> ```
> Образ `build-123`, ему 20 дней  
> ➡️ Он попадёт под удаление

**Можно отключить:**  
Не указывайте `NO_PREFIX_RETENTION_DAYS` в `.env`.  
➡️ Тогда образы без префикса не будут удаляться по времени, только по `MAX_RETENTION`, если он включён.

---

## 🤔 Поведение при отсутствии параметров

Если **оба параметра отключены** (`MAX_RETENTION_DAYS` и `NO_PREFIX_RETENTION_DAYS`), скрипт опирается только на `PREFIX_RULES`.

Если `prefix_rules` отсутствует в YAML-конфиге:
- Все теги считаются "без префикса".
- К ним применяется `no_prefix_retention_days`, если он задан.
- Если и он не указан — такие образы не удаляются вообще.

---

## 🛠 Пример конфигурационного YAML-файла

```yaml
repo_names:
  - test1
  - test2

prefix_rules:
  dev:
    retention_days: 0
    reserved: 1
  test:
    retention_days: 0
    reserved: 1

max_retention_days: 30        # Можно опустить
no_prefix_retention_days: 15  # Можно опустить
dry_run: true                 # true — только лог, без удаления
```


Если у тебя Nexus умеет удалять данные из MinIO , но не может удалить данные из NetApp S3 , то проблема, скорее всего, связана не с самим протоколом S3 (он стандартный), а с:

Различиями в реализации S3 API в NetApp и MinIO.
Политиками доступа (IAM / права) .
Версионированием объектов или блокировкой удаления .
Поддержкой определённых операций S3 в Nexus .
🔍 Возможные причины и как проверить
1. NetApp S3 требует явного указания версии при удалении
Если включено версионирование (Versioning) , то простое DELETE без указания versionId просто создаст "delete marker", а не удалит объект.

✅ Как проверить:
Убедись, что версионирование отключено , или
Убедись, что Nexus умеет удалять все версии объекта .
🔧 Решение:
Отправь запрос с заголовком versionId, либо отключи версионирование на стороне NetApp StorageGRID (если это возможно).

2. Права доступа (IAM Policy / S3 Policy)
Nexus использует access key + secret key для подключения к S3-совместимому хранилищу. Возможно, эти учетные данные не имеют прав на удаление в NetApp S3.

✅ Как проверить:
Проверь политики доступа в NetApp S3 — должен быть разрешён метод s3:DeleteObject, s3:DeleteBucketPolicy и т.п.
Попробуй выполнить DELETE через AWS CLI или Postman с теми же ключами, чтобы понять, возвращается ли ошибка AccessDenied.
🔧 Решение:
Обнови политику пользователя или роли в NetApp S3 так, чтобы она включала:

3. NetApp S3 имеет дополнительную защиту (Object Lock, Governance Mode)
Если включен Object Lock (например, WORM-policy), то объекты нельзя удалить до истечения срока хранения, даже администратором.

✅ Как проверить:
В интерфейсе NetApp S3 (StorageGRID) посмотри, включена ли функция Object Lock для бакета.
Если включена, то удаление невозможно до окончания lock-периода.
🔧 Решение:
Либо отключи Object Lock.
Либо продли срок хранения, либо получи права на принудительное удаление (в Governance Mode иногда можно).
4. Nexus не отправляет корректный запрос DELETE
Может быть, Nexus отправляет некорректный запрос (например, не поддерживает удаление версий или формирует URL неправильно). MinIO более лоялен к таким ошибкам, чем NetApp S3.

✅ Как проверить:
Включи логирование запросов в Nexus (или проксируй через tcpdump/Postman).
Сравни, какие запросы он отправляет при удалении из MinIO и NetApp S3.

Если Nexus не умеет работать с версиями или не поддерживает все фичи NetApp S3, потребуется:

Обновить Nexus.
Использовать кастомный скрипт удаления.
Или переключиться на другую систему управления репозиторием.
